Keycloak and OpenLDAP is a perfect balance

Byte January 19, 2026

Ever since the My first (kinda) business project, I liked making apps for my family. It's simple, LAN, or worldwide through port forwarding. Back then, Keycloak was sufficient for SSO, because most apps supported the Security Assertation Markup Language (SAML) or OpenID Connect (OIDC). But if I ever add more old apps that don't support OIDC or SAML, they might support LDAP, LDAP is another protocol that's used to quickly get information on something, whether its a printer, a person (commonly), a document, and much more. I usually use LDAP for login, it's fast, it's existed since the 1990's, and it can store alot of stuff. But there's a con... it's hard to use. To some it may not be, but you need these stupid LDIF files or whatever to make any change. These LDIF files are basically instructions for what to change on the server, like make an account or update the password policy. Even with phpLDAPadmin, which is a web tool made to simplify LDAP management, it's still hard to use. Anyway, enterprise products or less flexible things like Microsoft's Active Directory or "LLDAP" exist, Active Directory is mostly paid though, and LLDAP is way less customizable. Anyway, if you want, here's how to connect an OpenLDAP server to keycloak if you store your users under "ou=people,dc=example"

First, in the keycloak admin console, go to User Federation, Add new provider, then LDAP, or if you haven't added a provider yet, it should say something like "Add LDAP Providers"

Anyway, here's the configuration I used, by the way I ran Keycloak and OpenLDAP in the same docker compose with the same docker network, so "ldap" worked as a hostname for the OpenLDAP server:

UI Display Name:   ldap
Vendor: Other
Connection URL: ldap://ldap:389
Enable StartTLS: Off  
Use Truststore SPI: (default)
Connection Pooling: (default)
Connection timeout: (blank)
Bind type: simple
Bind DN: cn=admin,dc=example
Bind credentials: (your admin pass here)
Edit mode: WRITABLE  
Users DN: ou=people,dc=example
Relative user creation DN: (blank)
Username LDAP attribute: uid
UUID LDAP attribute: (default)
User object classes: inetOrgPerson, organizationalPerson
User LDAP filter: (blank)
Search scope: One Level
Read timeout: (blank)
Pagination: (default)
Referral: (blank)
Import users: On
Sync Registrations: On
Batch size: (blank)
Periodic full sync: Off
Periodic changed users sync: On
Changed users sync period: 30
Remove invalid users during searches: On
All the kerberos stuff I just left off.
Cache policy: DEFAULT
Enable the LDAPv3 password modify extended operation: Off
Validate password policy: Off
Trust Email: on
Connection trace: (default)

Then I realized something, if a user resets their password through keycloak, keycloak just stores it in plaintext on OpenLDAP, thats very bad. I doubt a server with like 5 people will get a data breach, but I like practicing. So I wrote an LDIF to make it so all password received get hashed by OpenLDAP itself, since keycloak won't do it (from what I know).

Here's the LDIF I wrote if you want it too:

dn: cn=config
changetype: modify
replace: olcPasswordHash
olcPasswordHash: {SSHA}

That enables SSHA hashing for each password, which is considered strong. Applying LDIF files to your server depends on what you're running it on (Docker, Windows, Linux, etc) and some other things I think exist but can't be bothered to think of right now, so just google "how to apply an ldif file to (your server information)" Anyway, then I realized something, after I applied the LDIF, I tested it with ldappasswd, which is a tool made for resetting peoples passwords on LDAP servers. And guess what? It worked! I sent a cleartext password, and OpenLDAP hashed it before storing it. But with keycloak, it was still storing it in plaintext! It took me 20 minutes, then I figured out the "Enable the LDAPv3 password modify extended operation" under Advanced settings was the problem. I honestly don't understand it much, but after enabling it, OpenLDAP started hashing passwords it got from Keycloak. Yay!

That's all for now, thanks for reading, and I hope I taught you something.
