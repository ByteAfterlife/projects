My first worldwide project (Mumble)

Byte November 29, 2025

Before I start, I'll just say, this is a huge milestone in my adventures, I am pretty proud of myself for making this work.

I woke up one day (yesterday), and as someone who loves open source and self hosting, I decided to run my own Mumble server, If you don't know what Mumble is, it's basically this server code that you run to make your own voice call server that others can talk on. Now I've run plenty of servers before that are only accessible on my LAN. Simplified, the way networking works (if you wanna know in more detail, read this post) is unless you change some settings on your router, any server your computer runs cannot be seen by the rest of the world, it can only be accessed on people that are on the same network (are connected to the same wifi as you or ethernet or whatever you use to connect) as you. If not, they cannot access it. But, today I decided is the day I make it accessible anywhere. Firstly, I setup the Mumble server itself with Docker, then I tested it. It works. Now was the hard part, the part I never did before. To those who have done this before, it may seem small, but this was my first time, it was really exciting! So I went into my parents router settings (they don't mind), and looked up how to setup port forwarding on my router model (it was pretty complicated) but eventually it worked! I had to make a "custom service," specify ports, connect it to my computer through some weird UI, but in the end, I did it! I got it so if you connect to port 64738 (default mumble port) through my public IP (the magical number the world uses to connect to your servers and router) it routes you to my computer's mumble server! Perfect.

But now here's another problem.. most home routers change your public IP, that's bad. By the way, this mumble server was just for me and my family to decentralise from corporations and whatnot. So if the public IP changed, my family would need to change it all on their computers too, that would be annoying and a waste of time. So I realized something. You know domains right? Like example.com, or example.org, well, those are kind of aliases for IP addresses, like google.com is pointing to 142.250.191.142 as of the time I'm writing this, 142.250.191.142 is google's public IP right now, don't believe me? Go open your browser, and type 142.250.191.142 as the URL. Magic right? Nope, just networking basics. So anyway, the thing is.. these aliases can change. So what if instead of asking my family to use the public IP of my router as the address to connect to the Mumble server and have to manually change it everytime my router changes it's public IP, what if instead, I just got a domain, and anytime the public IP changed, I update the IP the domain is pointing to. This is actually a popular concept known as "Dynamic DNS" which from what I understood does exactly what I needed. Well, here's the problem, domains cost money. But, theres a free service that allows you to get free subdomains. So I got all the puzzle peices now! I got a domain, my port forwarding setup, the mumble server works. Now I just need to setup the domain and everything should work, right? Well, you'll find out later. So I made the domain record, setup Dynamic DNS by making a quick script that gets my public IP every 15 minutes, and sends it to the domain, if it changed, the domain forwards to my new public IP, if not, nothing changes. That's basically dynamic DNS, here's my script if you want. I had to censor alot because it includes login credentials and other private stuff.

  #!/bin/bash
USERNAME="(secret)"
PASSWORD='(secret)'
DOMAIN="(secret, kinda).privatedns.org"
IP=$(curl -s https://ipv4.icanhazip.com)

curl -k "https://${USERNAME}:${PASSWORD}@(secret)/nic/update?hostname=${DOMAIN}&myip=${IP}"

Since I'm using linux, I setup something called a "cron" job (that is basically an automated task scheduler) to run this script every 15 minutes. And guess what? It worked. If my public IP ever changes, my family just needs to wait a max of 15 minutes before my computer detects it and the domain is realigned with my router's new IP.

Now, an even bigger thing I never done before, get an actual certificate! Certificates, in simple, encrypt your connection with websites, almost every uses them, this blog uses it, Google uses it, as I said, almost every website in the world does, it makes sure nobody except the website can see what you're sending to the website, like your name and password, and it makes sure nobody except you can see what the website sent back, like your bank balance or email address or phone number. Certificates can be made with a "custom CA" but devices only trust a specific list of CAs, so what's a CA, again I'm simplifying this but basically it's a trusted organization that issues these certificates, the most popular one for website certificates is Lets Encrypt, they're trusted very very widely. So, I put my domain into lets encrypt and basically asked them for a certificate, this is necessary because it encrypts the voice communication between the mumble server I made and my family so nobody can eavesdrop. Then they told me I need a http server to prove I own the server, which I also had to setup and port forward. But after a whole bunch of work, I got a certificate, I also decided to setup another cron job to renew the certificate everyday at 3am. That's the story of my Mumble server journey. Hope you learned something!
